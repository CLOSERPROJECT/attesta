#+TITLE: Appwrite Identity
#+SUBTITLE: Replace Mongo-backed auth and org data with Appwrite sessions, teams, labels, and invite flows while keeping org-admin in Attesta
#+DATE: 2026-03-18
#+KEYWORDS: attesta, appwrite, auth, sessions, teams, labels, invites, org-admin, migration

Summary
- Replace Mongo-backed auth, sessions, organizations, roles, invites, and password resets with Appwrite. Keep Mongo only for processes, progress, attachments, and other workflow data not covered by this task.
- Remove Attesta platform-admin behavior and `/admin/orgs`; bootstrap organizations and first org-admins in Appwrite Console, while `/org-admin/*` stays in Attesta and mutates Appwrite through the Go backend.
- Keep workflow YAML `organizations` and `roles` as validation-only config. Use Appwrite teams for org membership, user labels for accepted business roles, and team prefs for org role catalog plus org metadata.
- Migrate process actor identity from Mongo ObjectIDs to opaque IDs so existing history remains readable and new Appwrite users can appear in action lists, timelines, and DPP traces.
- Vendor a pinned Appwrite self-hosting stack under `deployment/` and update both `deployment/docker-compose.local.yaml` and `deployment/docker-compose.coolify.yaml` so they start Attesta with the required Appwrite services, volumes, and env wiring.

* DONE [#A] Freeze the new identity boundary and data model
Effort: M

Notes: This is the root-cause step. Do not bolt Appwrite onto the existing `primitive.ObjectID` auth model; that would leak Mongo assumptions everywhere and make the migration brittle.

** DONE [#A] Replace user invites and user lifecycle operations with Appwrite membership flows
Why: The user explicitly wants real invitation emails sent by Appwrite, triggered from Attesta UI, while user/org/role state must no longer live in Mongo.
Change:
- Remove `CreateInvite`, `LoadInviteByTokenHash`, `ListInvitesByCreator`, and `DisableUser` from the Mongo store path.
- Invite behavior in `POST /org-admin/users` with `intent=invite`:
  - backend reads the current user’s Appwrite session secret from `attesta_session`
  - backend calls Appwrite’s team-membership invite endpoint using a client-style authenticated HTTP request, not the API-key server-SDK membership create path
  - reason: Appwrite docs state client-side membership creation sends the email invite, while server-side membership creation auto-adds the member and skips the acceptance flow
  - membership roles carry `owner|member` plus `attesta-role:*` pending roles until acceptance
- “Invites I sent”:
  - derive from pending and accepted Appwrite team memberships instead of Mongo invite documents
  - render `pending`, `accepted`, and `expired` from membership status and expiry fields exposed by Appwrite
- `intent=delete_user`:
  - if target is in the org team, remove team membership
  - clear Attesta role labels on the Appwrite user
  - do not delete the Appwrite user account globally
- Existing-user invite behavior:
  - if the user is already an accepted member of the org team, update labels directly and do not send a redundant invite
  - if the email belongs to another org team, reject as cross-org conflict, preserving current single-org semantics
Tests:
- Add invite and lifecycle tests covering:
  - invite email path uses the client-style Appwrite flow
  - duplicate pending invite does not create a second invite
  - existing in-org user update skips invite email
  - cross-org invite is rejected
